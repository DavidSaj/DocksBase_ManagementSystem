# Track 9 — Charter, Boat Hire & Commercial Harbour: Design Spec
Date: 2026-05-07
Scope: New `charter` and `harbour` Django apps covering charter fleet inventory, charter bookings (wizard + skipper assignment + OTA webhook imports + agent commissions), charter agreements via Dropbox Sign, short-term boat rental (drag-and-drop calendar with public guest checkout), commercial vessel movements, harbour dues calculated from tariff tables, and harbour authority reporting.

> Spec status: Final — reviewed 2026-05-08

---

## 1. Architectural Goal

Track 9 introduces two self-contained Django apps that extend the existing platform without modifying its core. The `charter` app handles all hire of vessels to customers — from 15-minute electric dinghy sessions to multi-week skippered passages. The `harbour` app handles commercial shipping traffic at marinas that operate as working ports.

Both apps are optional at the marina level: `charter` is toggled by `marina.charter_enabled` and `harbour` by `marina.harbour_enabled`. The `harbour` feature is additionally gated behind an "Enterprise Port" subscription tier in the billing logic. Marinas that are pure leisure marinas with no commercial traffic will never see harbour screens. Every model in both apps carries the standard `marina = ForeignKey(Marina)` multi-tenancy key.

All fees generated by either app route through `billing.ChargeableItem` → `billing.InvoiceLineItem` → `billing.Invoice`. No pricing lives directly in charter or harbour model fields — rates are stored in `ChargeableItem` records (category: `charter` or `harbour_tariff`) and the tariff engine reads from them at calculation time.

The drag-and-drop rental calendar (§7.3) follows the same visual language as the Boatyard screen's storage grid. The charter fleet availability calendar follows a Gantt-style horizontal timeline, one row per vessel, bookings as coloured bars — the same mental model as a haul-out schedule.

OTA bookings from Zizoo and Click&Boat arrive via inbound webhook. The OTA webhook receiver and channel authentication are built in v1 (not deferred). When a booking webhook arrives, the system maps it to a `CharterVessel`, creates a `CharterBooking` with the appropriate `channel` value, and immediately blocks out the vessel on the Gantt calendar. Manual channel tagging is not a substitute.

Short-term rental bookings support a public guest-checkout flow (no DocksBase portal account required). On payment, the backend silently creates a shadow `Member` record for CRM use. `RentalBooking.member` is therefore nullable; the `customer_name`, `customer_email`, and `customer_phone` fields capture the walk-in customer details.

## 3. Data Models — Charter

### 3.1 `CharterVessel`

Extends `Vessel` with a `OneToOneField`. The parent `Vessel` record (in `vessels` app) holds all physical specs (loa, beam, flag, owner FK to `Member`, etc.). `CharterVessel` adds the commercial layer.

```python
class CharterVessel(models.Model):
    marina        = models.ForeignKey('accounts.Marina', on_delete=models.CASCADE, related_name='charter_vessels')
    vessel        = models.OneToOneField('vessels.Vessel', on_delete=models.CASCADE, related_name='charter_profile')

    # Rates — stored as ChargeableItem references (single source of truth)
    # Each FK points to a ChargeableItem with category='charter'.
    # The pricing_model on the ChargeableItem determines the unit (per_hour / per_night / per_week).
    hourly_rate_item  = models.ForeignKey('billing.ChargeableItem', null=True, blank=True, on_delete=models.SET_NULL, related_name='charter_hourly')
    daily_rate_item   = models.ForeignKey('billing.ChargeableItem', null=True, blank=True, on_delete=models.SET_NULL, related_name='charter_daily')
    weekly_rate_item  = models.ForeignKey('billing.ChargeableItem', null=True, blank=True, on_delete=models.SET_NULL, related_name='charter_weekly')
    cleaning_fee_item = models.ForeignKey('billing.ChargeableItem', null=True, blank=True, on_delete=models.SET_NULL, related_name='charter_cleaning')
    skipper_fee_item  = models.ForeignKey('billing.ChargeableItem', null=True, blank=True, on_delete=models.SET_NULL, related_name='charter_skipper')

    fuel_inclusive      = models.BooleanField(default=False)
    skipper_required    = models.BooleanField(default=False)
    min_charterer_qual  = models.CharField(max_length=200, blank=True, help_text='e.g. RYA Day Skipper or ICC')
    security_deposit    = models.DecimalField(max_digits=10, decimal_places=2, default=0)
    max_duration_days   = models.IntegerField(null=True, blank=True)

    # owner_type and commission_rate have been removed.
    # Ownership structure and revenue splits are modelled via CharterManagementAgreement
    # (see Section 3.1a) to support syndicate ownership and mid-season vessel sales.

    is_available     = models.BooleanField(default=True)
    notes            = models.TextField(blank=True)
    created_at       = models.DateTimeField(auto_now_add=True)

    class Meta:
        ordering = ['vessel__name']

    def __str__(self):
        return f'Charter: {self.vessel.name}'
```

**Constraint:** `hourly_rate_item`, `daily_rate_item`, `weekly_rate_item`, `cleaning_fee_item`, and `skipper_fee_item` must each have `category='charter'` when set. Enforce in serializer validation (same pattern as `BerthSerializer.validate_pricing_tier`).

### 3.1a `CharterManagementAgreement`

Replaces the removed `owner_type` / `commission_rate` fields on `CharterVessel`. Modelling ownership as a single static field on the vessel cannot support syndicate ownership (e.g. Owner A 40%, Owner B 60%) or mid-season vessel transfers — both are normal in high-end charter fleets. A sold or reassigned vessel would require mutating the historical `CharterVessel` record, corrupting payout calculations for past bookings.

`CharterManagementAgreement` instead tracks each ownership period explicitly:

```python
class CharterManagementAgreement(models.Model):
    """
    Represents one owner's share of a charter vessel for a defined validity window.
    Multiple active agreements for the same vessel are permitted — split_percentage
    values for all active agreements on a vessel must sum to exactly 100%.
    A marina-owned vessel has a single agreement with split_percentage=100 and
    member=None (or a nominated internal member).

    Mid-season transfers: close the existing agreement (valid_to = transfer_date - 1 day)
    and open a new one. Historical bookings retain their payout calculations via the
    agreement that was active at booking creation time (stored on CharterAgentCommission
    as commission_amount, which is now recomputed on subtotal changes — see Section 3.4).

    Payout calculation uses the agreement active at CharterBooking.start_dt.
    """
    marina           = models.ForeignKey('accounts.Marina', on_delete=models.CASCADE, related_name='charter_management_agreements')
    charter_vessel   = models.ForeignKey(CharterVessel, on_delete=models.CASCADE, related_name='management_agreements')
    member           = models.ForeignKey('members.Member', on_delete=models.SET_NULL, null=True, blank=True,
                                         related_name='charter_management_agreements',
                                         help_text='The vessel owner / syndicate member. Null = marina-owned.')
    owner_label      = models.CharField(max_length=200, blank=True,
                                         help_text='Display name for payout reports (e.g. "Smith Syndicate — 40%")')
    split_percentage = models.DecimalField(max_digits=5, decimal_places=2,
                                           help_text='This owner\'s share as a percentage (e.g. 40.00). '
                                                     'All active agreements for a vessel must sum to 100.')
    commission_rate  = models.DecimalField(max_digits=5, decimal_places=2, default=0,
                                           help_text='Marina commission rate retained from this owner\'s share (%)')
    valid_from       = models.DateField()
    valid_to         = models.DateField(null=True, blank=True, help_text='Null = current / open-ended agreement')
    notes            = models.TextField(blank=True)
    created_at       = models.DateTimeField(auto_now_add=True)

    class Meta:
        ordering = ['charter_vessel', 'valid_from']

    def __str__(self):
        return f'{self.charter_vessel.vessel.name} — {self.owner_label or "Marina"} {self.split_percentage}%'
```

**Invariant enforced at save:** the sum of `split_percentage` across all agreements for a `charter_vessel` where `valid_to IS NULL OR valid_to >= today` must equal 100. Enforce in serializer `validate()` with a clear error message. The admin can temporarily set one agreement to `valid_to = today` and create the new one in the same request to perform a clean transfer.

**Important:** A `Vessel` can be linked to a `CharterVessel` or a `RentalUnit`, but not both simultaneously. These are mutually exclusive fleet types. The data model enforces this at the database level — attempting to create both a `CharterVessel` and a `RentalUnit` for the same underlying `Vessel` must raise a validation error. If a marina wants to offer an electric day boat on both a walk-in hourly basis and skippered half-day trips, it must be registered as a `RentalUnit` and "full-day" slots blocked off manually.

### 3.2 `CharterBooking`

```python
class CharterBooking(models.Model):

    charter_vessel   = models.ForeignKey(CharterVessel, on_delete=models.PROTECT, related_name='bookings')
    charterer        = models.ForeignKey('members.Member', on_delete=models.SET_NULL, null=True, blank=True, related_name='charter_bookings')
    charterer_name   = models.CharField(max_length=200, blank=True, help_text='For non-member charterers')
    charterer_email  = models.EmailField(blank=True)
    charterer_phone  = models.CharField(max_length=30, blank=True)

    skipper          = models.ForeignKey('staff.StaffMember', on_delete=models.SET_NULL, null=True, blank=True, related_name='charter_assignments')

    start_dt         = models.DateTimeField()
    end_dt           = models.DateTimeField()
    duration_unit    = models.CharField(max_length=10, choices=DurationUnit.choices, default=DurationUnit.DAILY)

    # Snapshot of applied rates at booking time (ChargeableItem unit_price at time of booking)
    rate_applied     = models.DecimalField(max_digits=10, decimal_places=2)
    fuel_inclusive   = models.BooleanField(default=False)
    cleaning_fee     = models.DecimalField(max_digits=8, decimal_places=2, default=0)
    skipper_fee      = models.DecimalField(max_digits=8, decimal_places=2, default=0)
    deposit_amount   = models.DecimalField(max_digits=8, decimal_places=2, default=0)
    deposit_status   = models.CharField(max_length=20, choices=DepositStatus.choices, default=DepositStatus.PENDING)

    class DepositMechanism(models.TextChoices):
        AUTH_HOLD = 'auth_hold', 'Stripe Auth & Hold (< 7 days)'
        CAPTURED  = 'captured',  'Captured to Card + Credit Account Liability (>= 7 days)'

    deposit_mechanism = models.CharField(max_length=20, choices=DepositMechanism.choices,
                                          blank=True,
                                          help_text='Determined at booking creation based on charter duration. '
                                                    'Auth holds expire after 7 days; long charters use capture + credit account.')
    deposit_stripe_payment_intent = models.CharField(max_length=200, blank=True,
                                                      help_text='Stripe PaymentIntent ID. For auth_hold: capture_method=manual. '
                                                                'For captured: a completed charge; refunded on safe return.')

    subtotal         = models.DecimalField(max_digits=10, decimal_places=2, default=0)
    total            = models.DecimalField(max_digits=10, decimal_places=2, default=0)

    # OTA channel (populated for OTA-sourced bookings via webhook)
    channel          = models.CharField(max_length=50, blank=True, help_text='e.g. zizoo, click_and_boat, direct')
    channel_ref      = models.CharField(max_length=200, blank=True, help_text='OTA booking reference')
    channel_commission = models.DecimalField(max_digits=8, decimal_places=2, default=0)

    invoice          = models.ForeignKey('billing.Invoice', on_delete=models.SET_NULL, null=True, blank=True, related_name='charter_bookings')
    status           = models.CharField(max_length=20, choices=Status.choices, default=Status.ENQUIRY)
    internal_notes   = models.TextField(blank=True)
    created_at       = models.DateTimeField(auto_now_add=True)

    class Meta:
        ordering = ['-start_dt']
        # OTA webhook retries must not create duplicate charter bookings.
        # The constraint is partial (only applies when channel_ref is non-empty)
        # to avoid conflicts between multiple direct/non-OTA bookings with blank channel_ref.
        constraints = [
            models.UniqueConstraint(
                fields=['marina', 'channel', 'channel_ref'],
                condition=models.Q(channel_ref__gt=''),
                name='unique_ota_charter_booking_per_marina_channel',
            )
        ]

    def __str__(self):
        return f'Charter #{self.pk} — {self.charter_vessel.vessel.name} ({self.start_dt.date()})'
```

**Business rules:**
- `start_dt` must be before `end_dt`.
- If `charter_vessel.skipper_required` is True, `skipper` must be set before status can move to `confirmed`.
- Skipper validation: the assigned `StaffMember` must have a `Certification` record with a name matching the vessel's `min_charterer_qual` (or a master skipper cert) and `status='valid'`. Enforce in serializer.
- No two `CharterBooking` records for the same `charter_vessel` may overlap in time (except where one is `cancelled`). Enforce via `clean()` and a serializer-level overlap check.

**Deposit handling — 7-day Stripe authorization limit:**

Stripe card authorizations (uncaptured PaymentIntents) expire after exactly 7 days. A standard auth-and-hold strategy fails silently for any charter longer than one week: the hold evaporates mid-charter and the marina cannot capture for damages on return. The deposit mechanism is therefore determined by duration at booking creation:

**Charters < 7 days (`deposit_mechanism = 'auth_hold'`):**
- Create a Stripe PaymentIntent with `capture_method="manual"`. The card is authorized but not charged.
- `deposit_status` progresses: `pending` → `held` → `released` or `withheld`.
- On safe return: call `stripe.PaymentIntent.cancel()` — no charge to the charterer.
- On damage: call `stripe.PaymentIntent.capture(amount_to_capture=repair_amount)` — charged to the charterer's card.
- The UI must display a countdown banner in `CharterBookingDrawer` when the hold is within 48 hours of its 7-day expiry.

**Charters ≥ 7 days (`deposit_mechanism = 'captured'`):**
- Create a Stripe PaymentIntent with `capture_method="automatic"` (or confirm immediately). The deposit is physically charged to the charterer's card.
- Post the captured amount as a liability credit to the charterer's `MemberCreditAccount` (Track 4): `MemberCreditAccount.credit(amount=deposit_amount, description='Security deposit hold — Charter #{pk}')`.
- `deposit_status` progresses: `pending` → `held` (deposit captured and posted to credit account) → `released` or `withheld`.
- On safe return: issue a standard `stripe.Refund.create(payment_intent=..., amount=deposit_amount)` and debit the `MemberCreditAccount` liability entry.
- On damage: debit only the repair cost from the credit account; refund the remainder to the card.

**Why capture + credit account:** The captured deposit appears as a liability (money held but not earned) on the marina's GL. The `MemberCreditAccount` entry provides an auditable record that the money is being held as a refundable deposit, not revenue. This is the correct GAAP treatment for refundable security deposits.

### 3.3 `CharterAgreement`

Reuses the existing `documents.Envelope` infrastructure. `CharterAgreement` is a thin wrapper that ties a `CharterBooking` to an `Envelope`.

The booking wizard allows the staff member to select from a list of active `DocTemplate` records tagged as charter agreements. Multiple templates are supported to accommodate different agreement types (e.g. a 1-page liability waiver for a 2-hour electric boat rental vs. a full bareboat charter contract for a multi-week passage). Template selection is a required step in the wizard.

```python
class CharterAgreement(models.Model):
    marina          = models.ForeignKey('accounts.Marina', on_delete=models.CASCADE, related_name='charter_agreements')
    booking         = models.OneToOneField(CharterBooking, on_delete=models.CASCADE, related_name='agreement')
    envelope        = models.ForeignKey('documents.Envelope', on_delete=models.SET_NULL, null=True, blank=True, related_name='charter_agreements')
    signed_at       = models.DateTimeField(null=True, blank=True)
    charterer_ip    = models.GenericIPAddressField(null=True, blank=True)
    created_at      = models.DateTimeField(auto_now_add=True)

    def __str__(self):
        return f'Agreement — Charter #{self.booking_id}'
```

When creating a `CharterAgreement`:
1. The staff user selects a `DocTemplate` from the dropdown in the booking wizard (filtered to templates with category `'waiver'` or `'charter_agreement'`).
2. Call the existing Dropbox Sign send-for-signature flow (same path as existing `Envelope` creation in `documents` app).
3. Set `envelope.vessel = booking.charter_vessel.vessel` and `envelope.recipient = booking.charterer` (if a `Member`).
4. When the Dropbox Sign webhook fires `signature_request_signed`, update `agreement.signed_at` and `booking.status` to `confirmed` (if currently `enquiry`).

### 3.4 `CharterAgentCommission`

```python
class CharterAgentCommission(models.Model):
    class PaymentStatus(models.TextChoices):
        PENDING  = 'pending',  'Pending'
        APPROVED = 'approved', 'Approved'
        PAID     = 'paid',     'Paid'

    marina           = models.ForeignKey('accounts.Marina', on_delete=models.CASCADE, related_name='charter_agent_commissions')
    booking          = models.ForeignKey(CharterBooking, on_delete=models.CASCADE, related_name='agent_commissions')
    agent_name       = models.CharField(max_length=200)
    agent_email      = models.EmailField(blank=True)
    commission_rate  = models.DecimalField(max_digits=5, decimal_places=2, help_text='Percentage')
    commission_amount = models.DecimalField(max_digits=8, decimal_places=2)
    payment_status   = models.CharField(max_length=20, choices=PaymentStatus.choices, default=PaymentStatus.PENDING)
    paid_at          = models.DateField(null=True, blank=True)
    notes            = models.TextField(blank=True)
    created_at       = models.DateTimeField(auto_now_add=True)

    def __str__(self):
        return f'Commission — {self.agent_name} / Charter #{self.booking_id}'
```

**Commission recognition rules — tied to earned revenue, not projected subtotal:**

`commission_amount` must NOT be locked at booking creation. If the booking subtotal changes after creation (partial refund due to storm cancellation, week-two cancellation, damage dispute adjustment), the commission must be recomputed against the final recognized revenue — not the original estimate. Overpaying an agent on revenue the marina never received is a direct financial loss.

Implementation requirements:
1. `commission_amount` is computed at creation as `booking.subtotal × (commission_rate / 100)`, but is NOT frozen there.
2. A `post_save` signal on `CharterBooking` detects changes to `subtotal` and recalculates `commission_amount` for all linked `CharterAgentCommission` records where `payment_status = 'pending'`. Records already `approved` or `paid` must NOT be retroactively altered — if a subtotal changes after approval, fire a staff alert: "Commission already approved — manual review required for Charter #{pk}."
3. `payment_status` may NOT transition to `'approved'` until `booking.status == 'completed'`. Enforce in the serializer `validate()`: `if obj.booking.status != 'completed': raise ValidationError('Commission cannot be approved until the charter is completed.')` The UI `[Approve]` button must be disabled and show a tooltip until the booking reaches `completed`.

```python
# charter/signals.py
@receiver(post_save, sender=CharterBooking)
def recompute_commission_on_subtotal_change(sender, instance, **kwargs):
    if not instance.tracker.has_changed('subtotal'):
        return
    # Wrap in on_commit so the task reads the fully committed subtotal,
    # not a stale value that may still be mid-transaction.
    def _recalc():
        pending_commissions = instance.agent_commissions.filter(payment_status='pending')
        for commission in pending_commissions:
            commission.commission_amount = instance.subtotal * (commission.commission_rate / 100)
            commission.save(update_fields=['commission_amount'])
        approved_affected = instance.agent_commissions.filter(payment_status__in=['approved', 'paid'])
        if approved_affected.exists():
            send_alert(marina_id=instance.marina_id, alert_type='commission_review_required',
                       subject=f'Commission already approved — Charter #{instance.pk} subtotal changed')
    transaction.on_commit(_recalc)
```

(Use `django-model-utils` `FieldTracker` on `CharterBooking.subtotal` to detect changes.)

**No-overlap + turnaround constraint:** Same `rental_unit` cannot have two non-cancelled bookings that overlap in time *or* that leave insufficient turnaround time between them. A 10:00–12:00 booking and a 12:00–14:00 booking do not overlap at the database level, but a physical kayak still needs inspection, sanitisation, and lifejacket check before the next customer takes it out. The constraint is:

```
Booking A end_dt + timedelta(minutes=rental_unit.turnaround_minutes) <= Booking B start_dt
```

Enforce in `RentalBookingSerializer.validate()`:

```python
def validate(self, data):
    unit = data['rental_unit']
    start = data['start_dt']
    end   = data['end_dt']
    buffer = timedelta(minutes=unit.turnaround_minutes)

    # select_for_update() acquires a row-level lock on all competing bookings for this
    # unit before the overlap check. Without this, two concurrent POST requests can both
    # pass the 'no conflict' check and both commit — creating a double-booking.
    # Must be called inside an atomic transaction (DRF's default behaviour wraps
    # create/update in atomic when ATOMIC_REQUESTS=True or using transaction.atomic()).
    conflict = RentalBooking.objects.select_for_update().filter(
        rental_unit=unit,
    ).exclude(status='cancelled').exclude(pk=self.instance.pk if self.instance else None).filter(
        # Any booking whose [end + buffer, end + buffer] window overlaps [start, end]
        start_dt__lt=end + buffer,
        end_dt__gt=start - buffer,
    )
    if conflict.exists():
        raise serializers.ValidationError(
            f'Booking conflicts with an existing booking or its {unit.turnaround_minutes}-minute turnaround window.'
        )
    return data
```

The availability endpoint (`GET .../rental-bookings/availability/`) must return both occupied slots AND the turnaround windows as distinct blocks in the response so the calendar can render them visually.

**Guest checkout flow:** When `online_booking=True` and `member` is null, the system processes the payment via Stripe and then creates a shadow `Member` record using the supplied `customer_email` as the lookup key (upsert: if a member with that email already exists, link to the existing record rather than creating a duplicate). The shadow member is tagged for CRM use but does not receive a portal account invitation unless manually triggered by marina staff.

### 4.2 `HarbourTariff`

The tariff table is the master pricing reference for all harbour dues. Each row defines the rate for one due-type / vessel-type combination.

Harbour tariffs support both stepped GT bands with a fixed base fee per band and a continuous per-GT multiplier. This accommodates the full range of real-world port tariff structures: a port that charges a flat €500 for ships between 1000–2000 GT sets `base_fee=500` and `multiplier_fee=0`; a port that charges €0.15 per GT sets `base_fee=0` and `multiplier_fee=0.15`. Both can be combined. The `ChargeableItem.unit_price` field is not used for the primary calculation on harbour tariffs; `base_fee` and `multiplier_fee` on `HarbourTariff` are the authoritative rate fields. The linked `ChargeableItem` (with `category='harbour_tariff'`) exists for billing line-item categorisation and tax rate lookup only.

## 5. Tariff Engine (Harbour Dues Calculation)

The tariff engine is a service module at `harbour/services/tariff_engine.py`. It is invoked when a `CommercialMovement` reaches `status='departed'` (or manually via the UI). It produces one `HarbourDueInvoice` per applicable due type and assembles them into a single `billing.Invoice`.

### 5.1 Tariff Lookup

```
def get_tariff(marina, due_type, vessel_type, flag, gross_tonnage, date) -> HarbourTariff | None
```

Lookup order (most specific wins):
1. Match on `due_type` + `vessel_type` + `flag_state` = vessel flag + GT band containing `gross_tonnage` + `effective_from <= date <= effective_to`.
2. If no flag-specific row, fall back to `flag_state=''` (all flags).
3. If no type-specific row, fall back to `vessel_type='all'`.
4. Return `None` if still no match (the UI warns and the due is skipped from auto-calculation).

GT band: a row applies when `min_gt <= gross_tonnage < max_gt` (or either bound is null, meaning open-ended).

### 5.2 Calculation by Due Type

All harbour dues use the formula: `calculated_amount = tariff.base_fee + (tariff.multiplier_fee × quantity)`.

The `quantity` value depends on the due type:

| Due Type | Quantity | Source field |
|---|---|---|
| `harbour_dues` | `gross_tonnage` | `movement.gross_tonnage` |
| `pilotage` | `gross_tonnage × pilotage_distance_nm` | `movement.gross_tonnage`, `movement.pilotage_distance_nm` |
| `tug` | `tug_duration_hours` | `movement.tug_duration_hours` |
| `passenger_landing` | `passenger_count` | `movement.passenger_count` |
| `cargo_handling` | `cargo_weight_mt` | `movement.cargo_weight_mt` |

For due types where `multiplier_fee = 0` (flat-fee-per-band tariffs), `calculated_amount = tariff.base_fee`. For continuous per-GT rates where `base_fee = 0`, `calculated_amount = tariff.multiplier_fee × quantity`.

### 5.3 Invoice Assembly

```python
def calculate_and_invoice(movement: CommercialMovement) -> Invoice:
    """
    For each applicable DueType, look up the tariff and create a HarbourDueInvoice.
    Assemble all line items into a single billing.Invoice.
    Invoice.source_type = 'commercial_movement'
    Invoice.source_id   = str(movement.id)
    Invoice.member      = None  (harbour invoices are addressed to the ShippingAgent, not a Member)
    Invoice.shipping_agent = movement.shipping_agent  (FK for consolidated agency billing)
    """
```

Due types calculated:
- `harbour_dues` — always if movement has `gross_tonnage`
- `pilotage` — if `movement.pilotage_distance_nm` is not null
- `tug` — if `movement.tug_duration_hours` is not null
- `passenger_landing` — if `movement.passenger_count > 0`
- `cargo_handling` — if `movement.cargo_weight_mt` is not null

Each `HarbourDueInvoice` creates one `InvoiceLineItem` with:
- `description` = `f"{tariff.get_due_type_display()} — {movement.vessel_name}"`
- `chargeable_item` = `tariff.chargeable_item`
- `quantity` = the relevant quantity value
- `unit_price` = `tariff.base_fee + (tariff.multiplier_fee × quantity)` (total amount; quantity on the line item is 1)
- `tax_rate` = `tariff.chargeable_item.tax_rate`

### 5.4 Re-calculation — Credit Note + New Invoice (legally required)

**Why voiding an issued invoice is illegal:** Under European tax regulations (GoBD in Germany, French LPF, UK HMRC rules), a commercial invoice that has been generated and sent to a Shipping Agent cannot be silently deleted or overwritten. Doing so breaks sequential invoice numbering and will fail a financial audit. The `void/recreate` pattern previously described in this spec is therefore prohibited once an invoice has been issued.

**Correct procedure when a `CommercialMovement` is corrected post-invoice:**

```python
def recalculate_movement_invoice(movement: CommercialMovement) -> tuple[Invoice, Invoice]:
    """
    Called when a CommercialMovement is edited after its invoice has been generated.
    Returns (credit_note, new_invoice).

    1. The original Invoice is immutable — it is NOT deleted or voided.
    2. A Credit Note (billing.Invoice with invoice_type='credit_note') is generated
       to fully neutralise the original invoice:
         - credit_note.related_invoice = original_invoice  (FK linking the two)
         - credit_note.total = -original_invoice.total
         - All InvoiceLineItems are mirrored at negative quantities
         - credit_note.status = 'issued' (immediately — it is not a draft)
    3. A brand-new Invoice is generated from scratch using the corrected movement data.
    4. All three documents (original, credit note, new invoice) are preserved in the GL.
       The shipping agent sees: original charge + credit note + corrected charge.
    5. The entire operation is wrapped in transaction.atomic().
    """
    with transaction.atomic():
        original_invoice = movement.due_invoices.select_related('invoice').first().invoice
        if original_invoice is None or original_invoice.status == 'draft':
            # Invoice has not been issued yet — safe to regenerate directly
            movement.due_invoices.all().delete()
            return calculate_and_invoice(movement), None

        # Issue a credit note neutralising the original
        credit_note = issue_credit_note(original_invoice)

        # Recalculate from the corrected movement data
        new_invoice = calculate_and_invoice(movement)
        new_invoice.related_credit_note = credit_note
        new_invoice.save(update_fields=['related_credit_note'])

        return credit_note, new_invoice
```

**`billing.Invoice` model additions required** (additive migration):
- `invoice_type = CharField(choices=[('invoice','Invoice'),('credit_note','Credit Note')], default='invoice')`
- `related_invoice = ForeignKey('self', null=True, blank=True, on_delete=SET_NULL, related_name='credit_notes')` — links a credit note back to the invoice it neutralises

**UI impact:** `MovementDrawer` must display the full document chain: Original Invoice → Credit Note → Corrected Invoice, each with a document-type badge. The `[Edit Movement]` action after invoice issuance must present a confirmation dialog: "This movement has an issued invoice. Editing will generate a Credit Note and a new Invoice. Continue?"

## 6. OTA Webhook Integration

The OTA channel integration is built in v1. The webhook receiver handles inbound booking notifications from Zizoo and Click&Boat.

### 6.1 Webhook Endpoints

```
POST  /api/v1/charter/webhooks/zizoo/          Zizoo booking webhook receiver
POST  /api/v1/charter/webhooks/click-and-boat/ Click&Boat booking webhook receiver
```

Both endpoints are unauthenticated at the session level but verified via channel-specific HMAC signature headers. They do not require marina-scoped authentication.

### 6.2 OTA Adapter Pattern

The OTA integration follows the `OTAChannel` adapter pattern established in Track 7. Each OTA has an adapter module at `charter/ota/{channel_name}.py` implementing a common interface:

```python
def verify_signature(request) -> bool: ...
def parse_booking(payload: dict) -> CharterBookingData: ...
def map_vessel(ota_vessel_id: str, marina) -> CharterVessel | None: ...
```

When a webhook arrives:
1. Verify the HMAC signature using the channel's secret key (stored in marina settings).
2. Parse the payload into a `CharterBookingData` dataclass.
3. Resolve the OTA vessel ID to a `CharterVessel` (lookup via `channel_ref` mapping; warn via email to marina if no match found).
4. Create a `CharterBooking` with `channel` set to the OTA name and `channel_ref` set to the OTA booking reference.
5. The booking immediately blocks out the vessel on the Gantt calendar.
6. Trigger the charter agreement send if the marina has auto-send configured.

If the webhook represents a cancellation, look up the existing `CharterBooking` by `channel_ref` and update its `status` to `cancelled`.

- **Y-axis:** one row per `RentalUnit`; row header shows unit name, type, and colour swatch
- **X-axis:** hours of the selected day (default: today); navigable by day
- **Cells:** 30-minute slots. Booked slots shown as coloured bars (using `rental_unit.colour`) labelled with customer name. Immediately following each booked bar, a grey hatched buffer block is rendered for `rental_unit.turnaround_minutes` to represent the inspection/sanitisation window. The buffer block is non-clickable and cannot be dragged over.
- **Drag to create:** user drags across empty slots on a unit row to define `start_dt`/`end_dt`. The calendar enforces the turnaround constraint client-side: if the dragged range starts within `turnaround_minutes` of a preceding booking's end, the range snaps forward to the earliest valid `start_dt`. A confirmation popover appears with auto-calculated rate and a customer name field; `[Confirm]` POSTs `RentalBooking`
- **Click existing booking:** opens a popover with booking detail and `[Cancel]` / `[Mark Returned]` actions

State: `const [selectedDate, setSelectedDate] = useState(new Date())`

Hooks:
- `useRentalUnits()` — fetches `GET /charter/rental-units/`
- `useRentalBookings({ date })` — fetches `GET /charter/rental-bookings/?date={date}`

Rate calculation for the popover:
- Duration < 4 hours → hourly rate × hours (from `hourly_rate_item.unit_price`)
- Duration 4–6 hours → half-day rate (from `halfday_rate_item.unit_price`)
- Duration >= 7 hours → full-day rate (from `fullday_rate_item.unit_price`)
- Displayed to staff before confirmation; no client-side price calculation — call `GET /api/v1/charter/rental-bookings/availability/` which returns the applicable rate via a `rate_preview` field

The calendar also has a **Summary strip** above it showing today's totals: Units Out, Completed Today, Revenue Today (sum of `total` on completed `RentalBooking` records for the day).

### 8.1 Charter Endpoints


#### Charter Bookings
```
GET    /api/v1/charter/bookings/                   List (?status=, ?vessel=, ?date_from=, ?date_to=)
POST   /api/v1/charter/bookings/                   Create (wizard submit)
GET    /api/v1/charter/bookings/{id}/              Retrieve
PATCH  /api/v1/charter/bookings/{id}/              Update status, skipper, deposit_status
POST   /api/v1/charter/bookings/{id}/send-agreement/   Trigger Dropbox Sign envelope send
POST   /api/v1/charter/bookings/{id}/release-deposit/  Release or withhold deposit (triggers Stripe cancel/capture)
```

Key request shape for booking creation (wizard final step):
```json
{
  "charter_vessel": 12,
  "charterer": 45,
  "start_dt": "2026-06-10T09:00:00",
  "end_dt": "2026-06-14T18:00:00",
  "duration_unit": "daily",
  "skipper": 7,
  "doc_template": 3,
  "channel": "direct",
  "agent_name": "",
  "agent_commission_rate": 0
}
```
Server calculates `rate_applied`, `cleaning_fee`, `skipper_fee`, `subtotal`, `total`, `deposit_amount` from the `CharterVessel`'s linked `ChargeableItem` records. Server initiates the Stripe card authorisation for the deposit amount. Client does not send price fields.

Response includes `price_breakdown` object for display in confirmation step.

(Create is triggered via `send-agreement/` action above.)

## 9. `StaffMember` Changes

The `staff` app requires one additive change to support contractor skippers.

Add `is_contractor = models.BooleanField(default=False)` to `StaffMember`. Contractor staff members:
- Appear in the skipper selection dropdown in the booking wizard (because they have verified certification records).
- Are excluded from standard payroll and HR views (filter: `is_contractor=False`).
- Are included in all compliance-related queries (certifications, assignments, audit trail).

No other changes to the `staff` app are required by this track.
