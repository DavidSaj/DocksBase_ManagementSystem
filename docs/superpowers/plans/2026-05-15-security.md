# Security PR Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship four bundled security sub-features for the manager dashboard — TOTP MFA, per-marina IP allowlist, periodic email re-verification, and security audit log — while keeping the boater portal entirely frictionless.

**Architecture:** One new Django app `apps.security` houses everything. Two additive fields on existing models (`User.email_verified_at`, `Marina.require_mfa_for_managers`). Two new middleware classes. Login flow gains a 2-step variant. Frontend gets a new `SecurityCard` in Settings + a TOTP/enrollment step on the Login screen.

**Tech Stack:** Django + DRF + simplejwt (already in place). New deps: `pyotp`, `qrcode[pil]` (backend) + `qrcode` npm package (frontend).

**Reference spec:** `docs/superpowers/specs/2026-05-15-security-design.md` — the spec is authoritative on any ambiguity in this plan.

---

## Sequencing rationale

Four backend implementer dispatches in order — each is independently testable, and Task N's tests don't block Task N+1:

1. **MFA backend + login flow change** — biggest blast radius (touches the token endpoint). Land first so subsequent tasks can layer on a working login.
2. **IP allowlist** — middleware-only addition. No dependency on T1.
3. **Email re-verify** — middleware + `User.email_verified_at` field. Independent of T1/T2.
4. **Audit log + retroactive wiring** — needs T1/T2/T3 because it wires events from those flows; the api_keys retroactive wiring is independent.

Then a single frontend implementer dispatch (T5) for all UI: Login screen MFA step + SecurityCard composition.

Each backend task is TDD. Frontend is build-pass only (no Vitest, matching the existing convention).
